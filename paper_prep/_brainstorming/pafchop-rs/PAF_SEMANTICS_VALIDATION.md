# pafchop-rs PAF semantics validation

Date: 2026-06-22

## Decision

The pre-existing f16 chopped PAFs generated by `pafchop-rs` 0.1.0 are **not
valid for identity-sensitive sweepGA filtering**.

They are acceptable only for coarse coordinate/window-presence checks that do
not use per-row identity, divergence, edit distance, CIGAR, or `cs` content.
They must not be used to decide identity thresholds because the old
implementation linearly interpolated target coordinates, residue matches
(PAF column 10), and alignment block length (PAF column 11) across each query
chunk. Linear interpolation is not an alignment operation and cannot recover
the distribution of matches, mismatches, insertions, or deletions inside a row.

## f16 input audit

The f16 package referenced by
`paper_prep/_brainstorming/pedigree_whole_genome_sweepga_fastga_frequency16`
was generated in worktree `agent-2649`. The logged raw PAF paths were sampled
directly from that worktree:

- `PAN027mat_vs_PAN010_joint.sweepga_frequency16_many_many_j0.paf.gz`
- `PAN027pat_vs_PAN011_joint.sweepga_frequency16_many_many_j0.paf.gz`
- `PAN028mat_vs_PAN027_joint.sweepga_frequency16_many_many_j0.paf.gz`

For the first 1000 rows in each file:

| comparison | sampled rows | `cg:Z` | `cs:Z` | `NM:i` | `dv:f` | `de:f` |
|---|---:|---:|---:|---:|---:|---:|
| PAN027mat_vs_PAN010_joint | 1000 | 1000 | 0 | 0 | 1000 | 0 |
| PAN027pat_vs_PAN011_joint | 1000 | 1000 | 0 | 0 | 1000 | 0 |
| PAN028mat_vs_PAN027_joint | 1000 | 1000 | 0 | 0 | 1000 | 0 |

The sampled f16 PAFs therefore contain enough operation-level information in
`cg:Z` to split rows exactly for query/target coordinates, PAF column 10, PAF
column 11, and CIGAR-derived identity metrics. They do not contain `cs:Z` in
the sampled rows, so base strings cannot be emitted for those specific rows;
however, `cs:Z` clipping is implemented for future rows that contain it.

If a future input row lacks `cg:Z`, exact per-chunk identity cannot be recovered
from PAF columns alone. Aggregate PAF columns 10 and 11 describe only the whole
row and do not identify where matches, mismatches, insertions, or deletions
fall along the query. The repaired tool rejects such rows instead of
interpolating identity. If a future `cg:Z` uses ambiguous `M` operations rather
than explicit `=`/`X`, `cs:Z` is required; otherwise exact per-chunk residue
matches cannot be recovered and the row is rejected.

## Recomputed, copied, and dropped fields

The repaired implementation recomputes these required PAF columns for every
emitted chunk:

| Field | Status | Rule |
|---|---|---|
| 1 query name | copied | Stable sequence identifier. |
| 2 query length | copied | Full query sequence length remains unchanged. |
| 3 query start | recomputed | Query-axis chunk start. |
| 4 query end | recomputed | Query-axis chunk end. |
| 5 strand | copied | Alignment orientation remains unchanged. |
| 6 target name | copied | Stable target sequence identifier. |
| 7 target length | copied | Full target sequence length remains unchanged. |
| 8 target start | recomputed | Minimum forward-strand target coordinate consumed by the chunk. |
| 9 target end | recomputed | Maximum forward-strand target coordinate consumed by the chunk. |
| 10 residue matches | recomputed | Sum of clipped `=` operations in the chunk, or `cs:Z` match/equal operations when the CIGAR contains ambiguous `M`. `X`, `I`, and `D` do not count as matches. |
| 11 alignment block length | recomputed | Sum of clipped `M`, `=`, `X`, `I`, and assigned `D` lengths in the chunk. |
| 12 mapping quality | copied | Mapping-quality interpretation is row-level and not recomputed. |

Target coordinates follow PAF semantics on both strands: field 8 is always less
than or equal to field 9 in forward target coordinates. For reverse-strand
alignments, the implementation walks the CIGAR from original target end toward
original target start and emits the min/max forward coordinates consumed by the
chunk.

These optional tags are alignment-derived and are never copied stale:

| Tag | Status | Rule |
|---|---|---|
| `cg:Z` | recomputed | Clipped CIGAR is emitted for each chunk. |
| `cs:Z` | recomputed when present | Clipped `cs` operations are emitted when the input row contains `cs:Z`. |
| `NM:i` | recomputed when present | Edit bases: `X + I + D` lengths from explicit CIGAR, or equivalent `cs:Z` operations when ambiguous `M` is present. |
| `dv:f` | recomputed when present | Base edit divergence: `NM / alignment_block_length`, printed with six decimals. |
| `de:f` | recomputed when present | Gap-compressed divergence: `(X bases + I events + D events) / alignment_block_length`, printed with six decimals. |
| `df:i` | recomputed when present | Edit bases, retained for the f16 aligner tag observed in the raw PAFs. |

All other optional tags are copied only when `--keep-tags` is requested. This
preserves non-alignment metadata such as `tp:A` or `st:Z` without duplicating
or trusting stale identity-sensitive tags. Without `--keep-tags`, only repaired
alignment tags and `zp/zc/zl/zo/zm/zs/ze/zts/zte` provenance tags are emitted.
The `zm:Z` tag and `chunk_mode` summary column record whether the row was
split in historical `row-start` mode or absolute `query-grid` mode.

## Boundary semantics

Chunks are defined on the query axis. In the backwards-compatible `row-start`
mode, chunk boundaries start at each row's `q_start`; in `query-grid` mode,
boundaries are absolute query-coordinate grid lines at `0,N,2N,...` intersected
with the row interval. For example, `q_start=7`, `q_end=27`, and `--length 10`
emits `7-10`, `10-20`, and `20-27` in query-grid mode, so another row covering
`10-20` emits exactly the same query slice.

Query-consuming operations (`M`, `=`, `X`, `I`) are clipped by overlap with the
chunk's query interval. Target-only deletions (`D`) have no query span, so they
are assigned deterministically to the chunk whose interval starts at that query
position; a terminal deletion at the original query end is assigned to the final
chunk. This makes no-overlap chunks sum back to the original CIGAR-derived
identity totals.

## Validation run

The Rust test suite includes golden tests for:

- `M`, `=`, `X`, `I`, and `D` operation clipping, including the requirement
  that ambiguous `M` identity be resolved by `cs:Z`;
- reverse-strand target coordinate semantics;
- chunks ending inside CIGAR operations;
- recomputed PAF columns 10 and 11;
- clipped `cs:Z` output when `cs:Z` is present;
- refusal to chop rows without `cg:Z`;
- query-grid chunk boundaries for shifted rows, reverse-strand rows, mixed
  CIGAR operations crossing grid lines, and input-order deterministic
  per-record concatenation;
- property-style checks that no-overlap chunks sum to original identity
  metrics across mixed operation strings.

Build commands used for this validation:

```bash
cargo test --manifest-path paper_prep/_brainstorming/pafchop-rs/Cargo.toml
cargo build --release --manifest-path paper_prep/_brainstorming/pafchop-rs/Cargo.toml
```
